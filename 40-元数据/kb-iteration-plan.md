# 知识库迭代机制方案

> 灵感来源：[Andrej Karpathy 关于 LLM Knowledge Bases 的帖子](https://x.com/karpathy/status/2039805659525644595)

## 一、核心理念

把 LLM 当作"编译器"，把零散原始资料"编译"成一个由 LLM 维护的 markdown 知识库。

| 原则 | 含义 |
|------|------|
| **原料/产物分离** | `10-原始资料/` 放原始资料，`20-知识库/` 放 LLM 编译产物 |
| **LLM 是唯一作者** | 人几乎不直接编辑 wiki，全部由 LLM 增量维护 |
| **Obsidian 当 IDE** | 只用来看三层视图：原始 / wiki / 衍生可视化 |
| **索引替代 RAG** | 小规模下，LLM 自动维护的 INDEX + 摘要就够用 |
| **问答反哺** | 每次问答的输出再"归档"回 wiki |
| **Lint 健康检查** | LLM 定期扫 wiki：找矛盾、补缺失、发现新文章候选 |

## 二、Vault 架构

```
ObsidianVault/
├── 00-收件箱/              临时区
├── 10-原始资料/                原始资料（只读）
│   ├── clippings/         Web Clipper 抓取
│   ├── ai-daily/          Horizon 每日速递
│   ├── xhs/               小红书导入
│   ├── papers/            论文
│   └── images/            图片
├── 20-知识库/               ⭐ LLM 编译产物
│   ├── INDEX.md           总索引
│   ├── 概念/          原子概念
│   ├── 主题/            主题长文
│   ├── 人物/            人物卡
│   └── 摘要/           原文摘要
├── 30-产出/            问答/探索产物
│   ├── qa/
│   ├── slides/
│   └── charts/
├── 40-元数据/               机制元数据
│   ├── lint-reports/
│   └── compile-logs/
└── 90-日记/              个人空间
```

## 三、四大工作流

### 1. Ingest（摄入）
- 原始资料 → `10-原始资料/{分类}/`
- 人只负责收集，不整理
- frontmatter 标记 `compiled: false`

### 2. Compile（编译）
触发 `/编译`：
1. 扫描 `10-原始资料/` 找未处理资料
2. 提取概念 → 在 `20-知识库/概念/` 新建/更新原子笔记
3. 更新 `INDEX.md`
4. 标记 `compiled: true` + 时间戳
5. 写编译日志到 `40-元数据/编译日志/`

### 3. Q&A（问答）
触发 `/问答 <问题>`：
1. 读 `INDEX.md` 定位
2. 综合回答 → 存到 `30-产出/问答/`
3. 新洞察回流到 `20-知识库/概念/`

### 4. Lint（健康检查）
每周一 9:00 自动触发 `/体检`：
- 找矛盾数据
- 找孤儿笔记
- 补缺失字段
- 建议新 topic
- 报告到 `40-元数据/体检报告/`

## 四、与 Karpathy 原方案的差异

1. **靠 `realclaudian` 插件**而非 hacky 脚本 — 原生集成
2. **增加 `40-元数据/`** — 让机制本身可观测
3. **保留 `90-日记/`** — LLM 写 wiki，人写日记的分工
4. **增加 `/问答` 命令** — 把 Q&A 流程也固化下来
